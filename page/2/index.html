亚马逊棋项目的总体设计

By 许小平

发表于 2016-12-13

亚马逊棋开发主要包括了以下四个模块

交互界面

交互界面主要用于展示棋盘，走棋的过程，以及与人的交互。包括选择先后手，重新开始，撤销操作等。有的队没有开发撤销功能，导致人在移动的时候出错，没法撤销。这种情况下是判输的，所有我们开发了撤销这个功能模块。

这块采用MFC编写。我直接绘制了一个图片上去。


同时初始化了四个黑棋和四个白棋，通过棋盘的位置，划分出了10*10的棋格，并且用二维数组保存了每个棋格左上角的位置，用于之后绘制棋子和箭头。

然后使用了一个10*10的二维数组保存每个棋格的状态。每个棋格总共有四种状态：

  • 空 0
  • 白棋 1
  • 黑棋 2
  • 箭头 3

这个交互界面不是很难。所有简要的叙述一下。

对弈引擎

比赛的场景是这样的：

  • 先选择先后手，
  • 先手由电脑产生一步走棋
  • 对方在自己的电脑上按照先手的走棋，手动的移动一步棋到对应的位置
  • 后手自动生成一步棋
  • 先手在自己的电脑上按照先手的走棋，手动的移动一步棋到对应的位置

比赛就一直重复这个过程，是不是不太自动化。这估计就是一点乐趣吧，双方都参与其中，而不只是电脑在下。

因此在开发亚马逊棋的时候，也是按照这个模式来的，电脑下了，人下。

简单的说，这个对弈引擎，就是满足人机切换，控制对弈流程，以及自动调用AI走起。
其中也涉及到一些具体的问题如下：

  • 设置先后手
  • 支持用户提子、落子、放箭的操作
  • 判断落子是否满足规则
  • 判断棋局是否结束，以及胜负
  • 棋盘更新
  • 自动调用AI走棋
  • 支持撤销功能

其中任何一个环节有问题都是失败的。上一届就是由于判断落子是否满足规则这里出了一个bug，导致惨败。

搜索引擎

搜索和评估是亚马逊棋的核心部分。亚马逊棋的走法非常多，前面说过开始的时候就有2176种，如果搜索2层大概是2000*2000=4000000个节点，要是三层以上就破亿了，虽然越往后，生成的节点数量越来越少，但是越往后，使用的搜索层次就会增加。在一定的阶段会达到一个瓶颈节点，这时搜索会特别的慢，可能5、6分钟才会搜出一步棋，比赛规定一局超过15分钟，超过后，用时最多者败。

搜索引擎主要解决的问题如下：

  • 如何生成走法
  • 如果搜到最佳走步
  • 如何根据对局进度调整搜索层次
  • 如何减少搜索的节点数
  • 如何搜索的更快，消耗的资源更熟

亚马逊棋的搜索的基本实现不是很难，我在这使用的是改良过的apha-beta剪枝算法。搜索的结果也是比较好，但是还是比较耗时耗资源的。后面文章将专门讲解这一块。

评估函数

人在下象棋的时候，往往会推测，我如果下了这一步棋，对方可能会走哪些棋。按照这种思考的模式，向后面推演几步，然后选择一步对自己最有优势的一步走法。
那么问题来了，人是能够判断一步棋对自己是有利还是不利的，电脑没有思考能力，那电脑怎么判断这步棋对自己是好还是坏了，因此产生了评估算法。

人类的思维在计算机上实现是需要量化，然后建立一个模型。计算机根据这个模型，来判断走棋的好坏。模型最终归于一个数学公式。

亚马逊棋介绍

亚马逊棋是由阿根廷沃尔特 Zamkauskas 于 1988 年发明的双人抽象策略游戏,是圈地游戏家族中的 一员,和国际象棋是远亲。

亚马逊棋的棋盘由 n*n(比赛时 10×10)的格子组成,对弈双方各有 4 个皇后, 棋盘初始摆放位置如图 1 所示,横坐标由 10 个大写 英文字母 A~J 表示,竖坐标由阿拉伯数字1~10表示。

  • 10*10的棋盘上，每方有四个棋
  • 每个棋子可以在八个方向上任意行走，但不能穿过障碍；
  • 当轮到一方行棋时，此方只能而且必须移动四个棋子中的一个，并在移动完成后，由当前移动的棋子释放一个障碍，障碍的释放方法与棋子的移动方法相同；
  • 当某方完成某次移动后，对方四个棋子均不能再移动时，对方将输掉比赛；
  • 每次开局位于棋盘下方的玩家先手；
  • 整个比赛中双方均不能吃掉对方或己方的棋子或障碍。

走法还是很简单的，没有象棋，围棋那么复杂，但是亚马逊的第一步的走法就有2176种，如果层数超过两层，生成的走法数就会上亿。棋面过于复杂，解的空间又如此巨大，所以亚马逊棋天生就不适合人与人下，倒是比较适合电脑与电脑对战。

亚马逊棋的研究，包括了编程语言，算法思想，博弈思想等；随着机器学习的开展，亚马逊棋作为计算机博弈的一个项目正逐渐的被更广泛的熟知。
研究内容主要包括：棋局表示，博弈搜索，走法生成，节点裁剪，评估函数

模块系统演化

By 许小平

发表于 2016-11-23

模块系统演化

一：CommonJS

该规范的核心思想是允许模块通过 require 方法来同步加载所要依赖的其他模块，然后通过 exports 或 module.exports 来导出需要暴露的接口。(nodejs采用)

优点：

  • 服务器端模块便于重用
  • NPM 中已经有将近20万个可以使用模块包
  • 简单并容易使用

缺点：

  • 同步的模块加载方式不适合在浏览器环境中
  • 同步意味着阻塞加载，浏览器资源是异步加载的
  • 不能非阻塞的并行加载多个模块

二：amd

Asynchronous Module Definition(异步加载) 规范其实只有一个主要接口 define(id?, dependencies?, factory)，它要在声明模块的时候指定所有的依赖 dependencies，并且还要当做形参传到 factory 中，对于依赖的模块提前执行，依赖前置。

优点：

  • 适合在浏览器环境中异步加载模块
  • 可以并行加载多个模块

缺点：

  • 提高了开发成本，代码的阅读和书写比较困难，模块定义方式的语义不顺畅
  • 不符合通用的模块化思维方式，是一种妥协的实现

三：cmd

Common Module Definition 规范和 AMD 很相似，尽量保持简单，并与 CommonJS 和 Node.js 的 Modules 规范保持了很大的兼容性。

优点：

  • 依赖就近，延迟执行
  • 可以很容易在 Node.js 中运行

缺点：

  • 依赖 SPM 打包，模块的加载逻辑偏重

四：ES6 模块(逐渐统一)

EcmaScript6 标准增加了 JavaScript 语言层面的模块体系定义。ES6 模块的设计思想，是尽量的静态化，使得编译时就能确定模块的依赖关系，以及输入和输出的变量。CommonJS 和 AMD 模块，都只能在运行时确定这些东西。

优点：

  • 容易进行静态分析

  • 面向未来的 EcmaScript 标准
    缺点：

  • 原生浏览器端还没有实现该标准

  • 全新的命令字，新版的 Node.js才支持

以前装nginx都是用yum命令直接安装，一步到位。现在借着服务器迁移的机会，来次源码安装的实践。

安装版本选择

Nginx官网提供了三个类型的版本

  • Mainline version：Mainline 是 Nginx 目前主力在做的版本，可以说是开发版
  • Stable version：最新稳定版，生产环境上建议使用的版本
  • Legacy versions：遗留的老版本的稳定版

我选择了nginx-1.10.2的稳定版本，下载到/usr/local/src的目录下

编译

  1. 服务器环境

  1. 依赖包安装

    nginx在编译和安装的时候需要一些工具和第三包模块，因此先下载安装

  1. 编译配置项

nginx大部分常用模块，编译时./configure –help以–without开头的都默认安装。

  • –prefix=PATH ： 指定nginx的安装目录。默认 /usr/local/nginx
  • –conf-path=PATH ： 设置nginx.conf配置文件的路径。nginx允许使用不同的配置文件启动，通过命令行中的-c选项。默认为prefix/conf/nginx.conf
  • –user=name： 设置nginx工作进程的用户。安装完成后，可以随时在nginx.conf配置文件更改user指令。默认的用户名是nobody。–group=name类似
  • –with-pcre ： 设置PCRE库的源码路径，如果已通过yum方式安装，使用–with-pcre自动找到库文件。使用–with-pcre=PATH时，需要从PCRE网站下载pcre库的源码（版本4.4 - 8.30）并解压，剩下的就交给Nginx的./configure和make来完成。perl正则表达式使用在location指令和 ngx_http_rewrite_module模块中。
  • –with-zlib=PATH ： 指定 zlib（版本1.1.3 - 1.2.5）的源码解压目录。在默认就启用的网络传输压缩模块ngx_http_gzip_module时需要使用zlib 。
  • –with-http_ssl_module ： 使用https协议模块。默认情况下，该模块没有被构建。前提是openssl与openssl-devel已安装
  • –with-http_stub_status_module ： 用来监控 Nginx 的当前状态
  • –with-http_realip_module ： 通过这个模块允许我们改变客户端请求头中客户端IP地址值(例如X-Real-IP 或 X-Forwarded-For)，意义在于能够使得后台服务器记录原始客户端的IP地址
  • –add-module=PATH ： 添加第三方外部模块，如nginx-sticky-module-ng或缓存模块。每次添加新的模块都要重新编译（Tengine可以在新加入module时无需重新编译）
